# 分析层 PRD（V1）

## 1. 文档定位

本文档定义平台分析层的产品目标、边界、输入输出、核心流程、当前已实现能力与 V1 待实现能力。

本文档采用双清单方式组织：

- `当前已实现`：指当前代码库中已经落地并可运行的能力
- `V1 目标能力（待实现）`：指已明确纳入分析层 V1 范围、但尚未完全落地的能力

本文档面向两类读者：

- 产品、业务、决策相关角色：理解分析层的目标、价值、边界与当前成熟度
- 研发、测试、协作角色：理解分析层的模块结构、能力范围、后续补齐方向与验收标准

---

## 2. 产品定位

分析层负责将项目输入转化为可追溯的多源证据、结构化因子判断、维度结论与最终分析报告，并支持人工复核与版本化。

分析层不是一个单纯的抓取工具，也不是一个只会输出分数的黑盒分析器。它是平台的判断中台，负责把多源资料收集、沉淀为证据，并输出可以支撑决策的分析结果。

---

## 3. 目标与价值

### 3.1 核心目标

分析层 V1 的目标是建立一个可持续扩展的分析闭环：

- 多源采集
- 证据沉淀
- 结构化判断
- 人工复核
- 最终分析报告输出

### 3.2 面向决策人的价值

- 快速判断项目当前整体状态
- 识别核心风险与核心机会
- 确认是否值得继续投入策略与执行资源
- 理解结论背后的证据基础

### 3.3 面向客户 / 项目方的价值

- 以专业克制方式输出判断结果
- 解释结论来源，而不是只给黑盒分数
- 呈现当前项目在产品、社区、内容、链上等维度上的真实状态

### 3.4 面向内部分析与复核的价值

- 将分散资料统一沉淀为证据
- 将分析结果拆解到因子层、维度层
- 支持人工改单因子、重算与版本留痕

---

## 4. 分析层边界

### 4.1 分析层负责

- 输入接收与任务建立
- 多源采集
- 证据构建
- 因子分析
- 维度汇总
- 总体分析输出
- 最终分析报告生成
- 人工复核、重算、版本管理

### 4.2 分析层不负责

- 生成完整运营策略
- 执行排期与资源编排
- 活动、PR、KOL、社区执行动作
- 团队协作与项目管理
- 深度安全审计
- 完整链上市场画像
- 无边界全网爬虫

### 4.3 与策略层的关系

- 分析层回答：项目当前是什么状态、风险与机会在哪里、证据是否支撑该结论
- 策略层回答：在此基础上接下来该怎么做

---

## 5. 用户角色与使用场景

### 5.1 用户角色

- 决策人 / 负责人
- 内部分析师 / 复核人员
- 客户 / 项目方（阅读最终报告）

### 5.2 典型使用场景

1. 用户提交官网、Docs、Twitter、Telegram、Discord、合约等资料
2. 系统建立任务并进行多源采集
3. 系统沉淀证据并运行因子分析、维度汇总与总体报告
4. 内部人员复核、改单因子、重算
5. 输出工作台阅读版与对外分析报告

---

## 6. 输入与采集范围

### 6.1 支持的输入类型

- 官网 URL
- Docs / Whitepaper URL
- Twitter / X URL
- Telegram 群链接
- Discord 邀请链接
- 目标链
- 合约地址（可多个）
- 补充说明 / Notes

### 6.2 采集范围定义

#### 官网 / Docs / 白皮书

不再以单页作为采集单位，而以资料域快照作为采集单位。

- 官网：同域受控多页采集
- Docs：文档域受控多页采集
- 白皮书：全文 + 章节快照采集

当前采用页面数量上限控制采集范围：

- 官网资料域：最多 30 页
- Docs 资料域：最多 50 页

#### Twitter

支持两类路径：

- public fetch
- browser fetch

采集目标包括：

- 账号页或推文页内容
- 页面结构化摘要
- 互动信号
- 可见评论样本

#### Telegram

通过官方 bot 进入目标群组，进行窗口化采样：

- 群基础上下文
- 社区窗口摘要
- 结构指标
- 消息样本包
- 社区质量评估

#### Discord

通过官方 bot 进入目标服务器，进行频道筛选与窗口化采样：

- 有效讨论频道发现
- 频道消息采样
- 社区窗口摘要
- 结构指标
- 消息样本包
- 社区质量评估

#### 链上

当前以 EVM 为主，支持：

- 目标链选择
- 多合约输入
- 合约基础识别
- 合约角色识别
- 代码特征检测
- LP 自动检索 + 人工确认
- 单池指标与多池聚合

---

## 7. 证据体系

分析层不直接依赖原始抓取动作做判断，而是依赖结构化证据。

### 7.1 核心对象

#### Source

表示一个来源入口，例如：

- 官网
- Docs
- Whitepaper
- Twitter
- Telegram
- Discord
- 合约地址

#### Collection Run

表示某次采集执行记录，用于记录：

- 哪个来源被采集
- 成功 / 失败 / partial
- 采集量
- warning / blocked reason

#### Evidence

表示分析层真正使用的证据单元，例如：

- 官网页面
- Docs 页面
- 白皮书章节
- 推文详情
- 页面评估
- 社区窗口摘要
- 社区结构指标
- 社区样本包
- 社区质量评估
- 链上指标
- LP 指标

#### Versioned Report Context

表示某次分析输出对应的证据与结论快照，便于复核与版本比较。

### 7.2 核心原则

- 结论依赖证据，而不是依赖单次抓取动作
- 每个分析输出都应可追溯到对应来源与证据

---

## 8. 分析流程

分析层的标准流程定义如下：

1. 创建任务
2. 建立来源
3. 触发采集
4. 形成证据
5. 运行单因子分析
6. 聚合维度结果
7. 生成总体分析结果
8. 人工复核
9. 重算与版本化
10. 输出最终分析报告

### 8.1 流程原则

- 采集层可以持续增强
- 分析输出必须始终能追溯到具体证据
- 人工复核优先于黑盒自动决策

---

## 9. 分析输出

### 9.1 因子输出

每个因子至少包含：

- 分数
- 判断理由
- 风险点
- 机会点
- 证据引用
- 置信度
- 是否被人工复核

### 9.2 维度输出

每个维度至少包含：

- 维度分
- 一句话结论
- 主要拖分因子
- 主要支撑证据

### 9.3 总体分析输出

总体分析至少包含：

- 总分
- 风险等级
- 核心问题
- 核心机会
- 总体总结

### 9.4 最终分析报告

最终分析报告同时存在两种形态：

- `工作台阅读版`
- `导出 / 汇报版`

报告结构采用固定六段式：

1. 执行摘要
2. 综合判断
3. 维度概览
4. 关键问题
5. 关键证据
6. 结论与建议入口

### 9.5 输出目标

分析层的目标不是产出大量分数，而是输出一份可追溯、可解释、可复核的最终分析报告。

## 10. 当前已实现能力 vs V1 目标能力

### 10.1 输入与任务管理

#### 当前已实现

- 多输入创建任务
- 独立来源对象
- 任务列表、任务删除
- 前端基础任务工作流

#### V1 目标能力（待实现）

- 输入校验更清晰
- 任务创建后的自动编排更顺
- 任务快照读模型进一步统一

### 10.2 官网 / Docs / 白皮书采集

#### 当前已实现

- 官网 / Docs 已从单页升级到资料域受控多页采集
- 白皮书支持全文 + 章节快照采集
- 新增内容材料已进入内容相关因子分析

#### V1 目标能力（待实现）

- Docs 目录结构感更强
- 白皮书章节切分更稳
- 多语言 / 重复页处理更聪明
- 资料域内容质量标记更细

### 10.3 社交与社区采集

#### 当前已实现

- Twitter public fetch
- Twitter browser fetch
- Telegram 窗口化采样
- Discord 频道筛选与窗口化采样
- 社区结构指标、样本包、质量评估

#### V1 目标能力（待实现）

- Twitter 稳定性继续增强
- Telegram 历史窗口能力增强
- Discord 频道筛选和区域汇总更聪明
- 采集状态表达更统一

### 10.4 链上采集

#### 当前已实现

- EVM 目标链选择
- 多合约输入
- L1 合约基础识别
- L2 合约角色识别
- L2.5 代码特征检测
- LP 自动检索 + 人工确认
- 单池指标与多池聚合

#### V1 目标能力（待实现）

- 更强的合约角色覆盖
- 更多基础链上信号
- 项目级链上画像入口预留

### 10.5 分析引擎

#### 当前已实现

- 因子分析
- 维度汇总
- 总体分析输出
- 人工改单因子
- 重算与版本化
- 内容资料面对内容因子的增强输入

#### V1 目标能力（待实现）

- 跨来源一致性判断更强
- 官网 / 白皮书深度判断更强
- 置信度体系更清晰
- 冲突证据仲裁更稳

### 10.6 报告输出

#### 当前已实现

- 工作台阅读版最终分析报告
- 六段式结构
- 内容资料面概况

#### V1 目标能力（待实现）

- 导出 / 汇报版成熟化
- 图形化表达增强
- 对外报告语言进一步稳定

---

## 11. 验收标准

分析层 V1 的验收标准如下：

- 能创建项目分析任务，并挂载多源输入
- 能从官网 / Docs / 白皮书形成资料域快照
- 能从 Twitter / Telegram / Discord / 链上采集到可用于分析的证据
- 能将采集结果沉淀为结构化证据体系
- 能输出因子、维度、总体分析结果
- 能支持人工改单因子、重算与版本化
- 能生成最终分析报告
- 报告能解释结论来自哪些来源和证据

分析层验收关注的是分析闭环是否成立，而不是单个采集器在任意站点上的偶发成功率。

---

## 12. 非目标

分析层 V1 的非目标如下：

- 不负责生成完整运营策略
- 不负责项目执行排期
- 不负责活动编排与资源投放
- 不负责团队协同管理
- 不负责深度安全审计
- 不负责全量链上市场画像
- 不负责无限制全网爬虫

---

## 13. 收尾定义

分析层 V1 的目标，是形成一个可持续扩展的分析闭环：

- 多源采集
- 证据沉淀
- 结构化判断
- 人工复核
- 最终分析报告输出

在此基础上，策略层再承接分析层结果，继续完成“接下来该怎么做”的工作。
